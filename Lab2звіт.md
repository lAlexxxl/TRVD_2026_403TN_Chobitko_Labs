Звіт з лабораторної роботи 2

Тема проєкту: Вебдодаток для відстеження ігрового прогресу та списку бажаного — "GameVault".
Обґрунтування вибору технологічного стеку
Для реалізації проєкту обрано наступний перелік технологій:
Рівень	Технологія	Обґрунтування
Back-end	C# / .NET 8	Обрано через високу швидкість розробки, надійну типізацію та вбудовані інструменти для створення RESTful API.
Front-end	React	Дозволяє створити динамічний інтерфейс, що важливо для швидкої фільтрації та пошуку в ігровій бібліотеці без перезавантаження сторінок.
Database	MongoDB	NoSQL підхід ідеально підходить для каталогу ігор, де структура даних (жанри, платформи, метадані) може бути гнучкою та часто змінюватися.
DevOps	Docker	Забезпечує легке розгортання бази даних та серверної частини в ізольованих контейнерах.

3. Проєктування бази даних (NoSQL Schema)
Для зберігання даних обрано документоорієнтовану модель (NoSQL). Нижче наведено схему основних колекцій та їхній зв'язок через Referencing (посилання за ID).

Приклад структури документа для колекції UserLibrary:
JSON
{
  "_id": "65f123abc...",
  "userId": "65f456def...", 
  "gameId": "65f789ghi...", 
  "status": "In Progress",
  "rating": 9,
  "notes": "Проходжу на Arch Linux, політ нормальний"
}
Обґрунтування: Використано стратегію посилань (Referencing) між користувачами та загальним каталогом ігор, щоб уникнути дублювання великих обсягів описів ігор у кожного окремого користувача.
Висновок
У ході лабораторної роботи №2 було спроєктовано архітектуру системи «GameVault» за стандартом C4. Визначено контейнери системи та протоколи їхньої взаємодії (REST API через HTTPS). Також розроблено схему бази даних MongoDB, яка враховує специфіку ігрового контенту та забезпечує гнучкість масштабування проєкту.
